A Philosophical Detour: The Intentional Stance

The Old Way: Syntactic Engine

We treat it as a machine following rules.

Input -> Rules -> Output

We must provide perfect, machine-readable instructions.

The New Way: Intentional System

We treat it as a rational agent with goals.

Goal -> Beliefs -> Action

We can explain our goals in plain language.

By adopting an intentional stance, we unlock a more powerful and natural way to interact with technology.

Addressing the Culture: The "Big Brother" Problem

Potential Issues

  • Chilling Effect: Will teams fear brainstorming?
  • Data Onboarding: How to ingest data non-intrusively?
  • Alert Fatigue: How to avoid false positive noise?

The "What, Why, How" Framework

  • WHAT: "We're introducing an automated security design assistant."
  • WHY: "To help you build faster and safer, reducing rework. This is a safety net, not surveillance."
  • HOW: "It works transparently by comparing design patterns to established policy. The goal is to assist, not report."

Implementation Models

Active Integrations: The Journey

These models actively assist teams during the design and planning phases.

  • 1
    Manual Check: A web tool where users paste plans for analysis.
  • 2
    ChatOps Assist: A bot in Slack/Teams for self-serve reviews.
  • 3
    CI Quality Gate: An automated check on "Planning-as-Code" repos.

Passive Analysis: The Destination

The most mature state, where analysis happens ambiently with little human interaction.

Requires: A rigorous security culture and two technical pillars:

1. Machine-Accessible Plans: Designs are formalized in frameworks like Markdown, ADRs, or internal schemas.

2. Rich Policy Corpus: Security policies are well-documented and available in a datastore for the AI to reference.

Active Mode: Planning-as-Code

To enable active enforcement, we treat planning documents like source code: versioned, reviewed, and checked in a CI/CD pipeline.
